## Context

ADR-019 D1 split the MCP bridge out of Phase 6 with the reasoning that
Activity Stream was pain-driven (ADR-018 lesson 3) while MCP was
expansion-driven. Phase 6 closed on schedule; this ADR brings the bridge
back as its own scoped phase.

**What an MCP bridge unlocks:** chat with your vault from Claude Desktop
or Claude Code without leaving the host app. Today, every Sagittarius
tool call requires opening Obsidian → focusing the chat panel → typing.
With MCP, you can ask Claude Desktop "what did I write about the Q3
roadmap last week?" and it queries the same vault via the same `search_vault`
tool that Sagittarius uses internally.

**What it doesn't unlock:** new capabilities. Every MCP-exposed tool already
exists in Sagittarius. This phase is plumbing, not new features.

**The architectural shift:** today Sagittarius is a *client* of the
Anthropic API. With MCP, it also becomes a *server* — its tools must be
reachable from outside the Electron renderer that hosts them. That's a
real attack-surface expansion and the auth/transport choices below are
load-bearing.

---

## Decisions

### D1 — Tool scope: read-side, write-side, or both

Three options:

**(a) Read-only — five tools (recommended for v0.9.0).** Expose
`read_note`, `list_folder`, `search_vault`, `get_backlinks`,
`get_graph_neighborhood`. No diff card needed because nothing mutates.
External clients can answer questions; they can't change the vault.
Lowest blast radius if auth leaks.

**(b) Read + write — fourteen tools.** Add the nine Phase 4 write tools.
Every external-driven write still routes through the Phase 4 diff card
(per ADR-016 D2), so the user's approval surface is unchanged.
**Caveat:** if Obsidian isn't focused, the diff card pops in the
background and external calls hang until the user finds it. UX risk.

**(c) Read-only at first, write-side as a v0.9.1+ slice once we've
worn the read side.** Lets us validate the transport, auth, and
activity-stream integration without write-blast-radius.

`<DECISION D1: PROPOSED — (c) read-only v0.9.0, write-side deferred to v0.9.x. Thad: ACCEPT / override>`

---

### D2 — Transport

Three MCP transports exist; only one fits an Obsidian plugin:

**(a) stdio.** Claude Desktop spawns the MCP server as a child
process. **Doesn't fit** — Obsidian is already running; we can't be
spawned by another app. Eliminated.

**(b) HTTP/SSE on localhost (recommended).** Sagittarius runs an
HTTP server inside the Electron renderer on a configurable port
(default `8765`). Claude Desktop / Claude Code MCP config points at
`http://localhost:8765/mcp` with a bearer token.

**(c) WebSocket on localhost.** Same shape as (b) but with WebSocket
upgrade. More complex; only valuable for streaming, which we don't
need.

`<DECISION D2: PROPOSED — (b) HTTP/SSE on localhost. Thad: ACCEPT / override>`

---

### D3 — Auth

External clients need a credential. Options:

**(a) Bearer token, generated on first enable (recommended).** Plugin
generates a 32-byte random token, shows it once in settings (with a
copy button), the user pastes it into the external client's MCP config.
Token is hashed at rest; raw value never written to disk after the
initial reveal. Regenerate command available.

**(b) No auth — localhost-only binding.** Anything on the user's
machine can call. Trivial to attack via any other Electron app or
malicious npm package that runs on the same machine.

**(c) Per-call OAuth handshake.** Overkill for a single-user local tool.

`<DECISION D3: PROPOSED — (a) bearer token, copy-once-then-hashed. Thad: ACCEPT / override>`

---

### D4 — Network exposure

Two options for the listening interface:

**(a) Localhost only — `127.0.0.1` (recommended).** No external machine
can reach the server. Removes a whole class of "I forgot the port was
open" mistakes.

**(b) All interfaces — `0.0.0.0`.** Lets Claude.ai web or a mobile app
on the same LAN reach the server. Powerful but requires the user to
think about firewalls and TLS. Out of scope for v0.9.0.

`<DECISION D4: PROPOSED — (a) localhost only. Thad: ACCEPT / override>`

---

### D5 — Activity stream integration

Every Sagittarius tool already emits to the activity log (Phase 6 PR #56).
MCP-driven calls should too, with an explicit attribution so the user
can distinguish "Claude Desktop did this" from "I did this in the chat
panel."

**Proposal:** extend the v0.8.0 event types with an optional `source`
field. Default `'sagittarius'`; set to `'mcp:<client>'` when the tool
was invoked over the MCP transport. Client is the `User-Agent` or
client-id from the MCP handshake (e.g. `mcp:claude-desktop`).

The ActivityView surfaces this via a new filter chip (or a tag on each
row); the spec for the filter chip lives in this decision and ships
in the v0.9.0 slice.

`<DECISION D5: PROPOSED — `source` field on every event; new `MCP` filter chip in ActivityView. Thad: ACCEPT / override>`

---

### D6 — Settings

New settings section "MCP bridge (Phase 6.5)":

| Setting | Default | Notes |
|---|---|---|
| `mcpEnabled` | `false` | Master switch. Off = no server, no port bound. |
| `mcpPort` | `8765` | Localhost port. Conflict-safe range; collision = surfacing a Notice + refusing to start. |
| `mcpToken` | (generated on first enable) | Hashed at rest. Visible once via "Reveal token" button (one-time, then re-hashes). |
| `mcpAllowedClients` | `[]` | Optional allowlist of MCP `clientInfo.name`. Empty = any authenticated client. |

`<DECISION D6: PROPOSED — four settings as above; off by default. Thad: ACCEPT / override>`

### D7 — Rollout slicing

Three-version slice:

| Slice | Adds | Notes |
|---|---|---|
| **v0.9.0** — MVP | HTTP/SSE server in Electron renderer, bearer auth, five read-only tools registered via MCP protocol, settings UI, activity emission with `source:` field | One PR per substrate: server, MCP protocol adapter, tool registration, integration. ~4-5 PRs total. |
| **v0.9.1** — Polish | `MCP` filter chip in ActivityView, "Reveal token" / "Regenerate token" buttons, port-conflict handling, smoke-test command (`Sagittarius: Test MCP connection`) | Smaller; closes the UX gaps from MVP |
| **v0.9.2** — Phase 6.5 close | Phase 6.5 retrospective ADR + release. Decision point: ship write-side as v0.9.x (per D1 (c)) or defer further. | Mirrors the v0.6.2 / v0.8.2 pattern |

`<DECISION D7: PROPOSED — three-slice rollout v0.9.0 → v0.9.2. Thad: ACCEPT / override>`

---

### D8 — Library choice

The plugin doesn't currently use any HTTP-server library. Two options:

**(a) `@modelcontextprotocol/sdk` (recommended).** Official SDK from
Anthropic. Bundles the MCP protocol, the HTTP/SSE transport, and the
JSON-RPC framing. Adds ~200KB to the bundle. Maintained against the spec.

**(b) Hand-roll on top of Node's `http` module.** Smaller bundle but
we own the MCP protocol implementation, which evolves. ADR-014's
HF SDK lesson (Phase 3 close) said it directly: "either bundle the
official SDK so protocol evolution is the SDK's problem, or version-pin."

`<DECISION D8: PROPOSED — (a) `@modelcontextprotocol/sdk`. Thad: ACCEPT / override>`

## Tool surface (binding for v0.9.0)

Five tools exposed over MCP (matching D1 (c)):

| Tool | Input | Output | Notes |
|---|---|---|---|
| `read_note` | `path: string` | `{ content, mtime, hash }` | Already exists; no changes |
| `list_folder` | `path: string` | `{ files, folders }` | Already exists |
| `search_vault` | `query: string, k?: number` | `{ hits: Citation[] }` | Already exists; uses Phase 3 retrieval |
| `get_backlinks` | `path: string` | `{ backlinks: string[] }` | Already exists |
| `get_graph_neighborhood` | `path: string, depth?: number` | `{ nodes, edges }` | Already exists |

No new tool definitions. The MCP adapter wraps the existing
`ToolRegistry` entries. v0.9.x writes would expand this without
reshaping the contract.

`<DECISION D9: PROPOSED — five read-only tools, no new tool definitions. Thad: ACCEPT / override>`

## Architecture sketch

```
src/
  mcp/
    McpServer.ts         # JsonRpcServer wraps @modelcontextprotocol/sdk
    McpToolAdapter.ts    # Bridges ToolRegistry → MCP tool schema
    auth.ts              # Token generation + hash verification
  main.ts                # Wire McpServer on plugin load when mcpEnabled
```

Lifecycle: `onload()` constructs the server (off by default); flipping
the settings toggle starts/stops the server without reload. Server
shares the existing `ToolRegistry` — same tool implementations, same
budget tracker, same activity log emission.

Auth: HTTP handler reads `Authorization: Bearer <token>` header,
compares against the hashed settings value via timing-safe compare.
Mismatch → 401. No token → 401. Match → JSON-RPC dispatch.

Threat model: localhost-only + bearer token + timing-safe compare.
Acceptable for a single-operator dev tool. Not acceptable for
multi-tenant or untrusted-network deployment — that's out of scope
for v0.9.x.

## Risks

| Risk | Mitigation |
|---|---|
| Port collision with another local service | Detect EADDRINUSE on start; surface a Notice with the offending port; user picks a new port in settings |
| Token leaks via clipboard / git accidentally | Hash at rest + one-time reveal + `Regenerate token` button; document in settings description |
| MCP spec churn — protocol revs faster than the plugin | Pin SDK version; bump deliberately in retrospective ADRs (per ADR-014 lesson) |
| Diff card blocks external write call indefinitely | Out of scope for v0.9.0 (read-only); but for v0.9.x: server-side timeout on `Promise<approval>` → MCP returns `rejected: timeout` |
| External client gets a stale view if the vault changes mid-call | MCP is request/response; no streaming or watchers in v0.9.0. Future ADR if needed |
| Local-other-app exploit (malicious npm package on the same machine) | Bearer-token guarded. Not perfect — anything that can read settings can call the server. Accept; document in security docs |

---

## Out of scope (Phase 7+)

- Remote/LAN access (would need TLS + stronger auth)
- Multi-vault MCP routing
- MCP streaming notifications (live activity feed over MCP)
- Read-write tool exposure (D1 (c) defers this to v0.9.x; revisit in retrospective)
- Browser-based clients (CORS, more complex auth)
