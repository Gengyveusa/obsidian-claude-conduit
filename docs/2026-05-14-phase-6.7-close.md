## Context

Phase 6.7 opened with ADR-025 on the morning of 2026-05-14 (D1-D10
batch-accepted) and closes the same day with v1.1.0. The phase
shipped in three deliberate slices across three PRs, each a
complete behavior change at the version boundary:

- **v1.0.8 (PR #82) — substrate.** Added `source?: string` to
  `Transaction` per D5. Plumbed through `TransactionLog.begin(sessionId?,
  source?)` and `WriteToolContext.begin(sessionId?, source?)` to
  `write.committed` activity events. Added 5 settings keys
  (`mcpWriteEnabled`, `mcpWriteAllowedClients`, `mcpWritePathPrefixes`,
  `mcpWriteRateLimitPerHour`, `mcpHighRiskToolsEnabled`) with the
  "MCP write-side (Phase 6.7)" settings section. **No tool exposure
  yet** — in-app behavior unchanged. 8 new tests.

- **v1.0.9 (PR #83) — exposure + gates.** Made `MCP_EXPOSED_TOOL_NAMES`
  dynamic via `mcpExposedToolNames({ writeEnabled, highRiskEnabled })`.
  Built `src/mcp/WriteGate.ts` running five layered checks: master
  toggle (D1), high-risk (D1), per-client allowlist (D6), path-scope
  (D7 — both ends of `move_note`/`rename_note`/`link_notes`), rate
  limit (D9 — `WriteRateLimiter` rolling 60-min window). `McpHandler`
  opens transactions tagged `source: 'mcp:<client>'` so successful
  writes carry full provenance per D5. Concurrent in-app chat + MCP
  write yields "retry shortly" — simplest correct behavior. 31 new
  tests (22 WriteGate + 9 McpHandlerWriteSide).

- **v1.1.0 (PR #84) — queue + side panel + status bar pill + OS
  notifications.** New `ExternalProposalQueue` with promise-based
  `enqueue/respond` contract. `CallbackApprovalGate` routes by
  `WriteToolContext.currentSource()` — external proposals enqueue,
  in-app proposals stay on the diff-card callback. `McpHandler`
  races `registry.execute` against `mcpWriteQueueTimeoutMs` (D2 (c)
  hybrid, default 30 s); on timeout returns `queued` while the
  underlying tool keeps running in the background. New
  `ExternalProposalsView` with inline diff previews + Approve/Reject
  buttons; new status bar pill ("Sagittarius: N pending"); native
  Electron `Notification` on enqueue (opt-out via
  `mcpWriteNotifyOnQueue`). 21 new tests.

Test count grew from 869 (Phase 6.7 start) to 931 across all three
slices. New files: `src/mcp/WriteGate.ts`, `src/writes/ExternalProposalQueue.ts`,
`src/views/ExternalProposalsView.ts`. The plugin's `onunload` now
calls `queue.clearAll('plugin unloading')` so pending MCP `tools/call`
promises resolve with a clean rejection instead of dangling forever.

**OQ1 settled inline.** ADR-025 OQ1 asked "queue-view Approve — diff
card or skip?" The answer is **skip**: the side-panel row renders
the full proposal diff inline, so bouncing through a re-opened
modal would be redundant friction. If the answer turns out wrong,
it's one method call away from being flipped.

## The two lessons of Phase 6.7

### 1. Three-slice cuts across architectural layers beat splitting within a layer

ADR-025 was a 10-decision ADR with strong substrate (D5),
behavior (D1, D6, D7, D9), and UX (D2 (c), D3, D4) layers. The
slice decision was made explicitly at planning time: v1.0.8 is the
substrate, v1.0.9 the behavior, v1.1.0 the UX. Each PR was 300 /
1300 / 1400 LOC respectively and merged the same day. Compare to
Phase 7 (ADR-024 close), which shipped seven PRs across four
sub-versions because each version's "headline change" was a single
new rule. Phase 7 felt larger than Phase 6.7 even though both
shipped a similar volume of code, and the reason is that Phase 7's
slices were *within* the rules layer (one rule per PR) while
Phase 6.7's slices were *across* layers (substrate → behavior → UX).

The architectural shape mattered:
  - **Slice 1 was risk-free** because it changed zero user-visible
    behavior. Pure type plumbing + settings keys. If it broke
    something, the blast radius was unit-test coverage.
  - **Slice 2 was reviewable** because Slice 1 had already shipped
    the field on `Transaction`. The "where does source come from?"
    question was already answered; Slice 2 only needed to *set* it
    correctly in McpHandler.
  - **Slice 3 was almost entirely UI** — the architectural decisions
    were settled by Slices 1+2. Reading PR #84's diff feels like a
    React refactor, not a load-bearing redesign.

**Lesson:** when a phase plan has clean substrate / behavior / UX
layers in its decisions, prefer slicing across the layers over
slicing within them. The visible "behavior change" per version is
larger, reviews are scoped to one architectural concern, and any
slice you ship on green is a stable rest point.

(Corollary: when there isn't a clean layer split — e.g. "ship more
rules" — within-a-layer slicing is the right answer because no
across-layer cut exists.)

### 2. Solve the realistic case well; let the corner case wait for evidence

The diff-card-focus problem (ADR-023 lesson 2) has two faces:
  - **(a) user not at Obsidian.** MCP client asks for a write while
    the user is in another workspace, on another screen, away from
    the desk.
  - **(b) in-app chat concurrently mid-turn.** ConduitAgent.chat()
    has opened the singleton `WriteToolContext`; MCP request can't
    open another transaction on the same context.

ADR-025 D2 (c) targeted (a) with the queue + timeout race. Slice
3 ships that fully. For (b), Slice 2 chose the simplest correct
behavior: catch the `begin()` throw, return "In-app chat is in
progress, retry shortly." I was tempted multiple times to design
a per-call `WriteToolContext` or `AsyncLocalStorage`-based context
so (b) would Just Work — but that refactor would have touched
every write tool's `apply()` path (10 tools), changed the deps
shape of every tool factory, and rewritten the `WriteToolContext`
contract. Cost: probably another 500-LOC PR. Realistic frequency:
the user has to be chatting *and* an MCP client has to fire *at
the same time*, which the user controls.

The "retry shortly" path is honest about the constraint and lets
the operator handle it (wait 5 seconds and try again). If it
turns out to bite real workflows, that's the moment to write the
ADR for the per-call-context refactor. Until then, the corner
case stays a corner.

**Lesson:** when an architectural problem has multiple facets,
solve the most-frequently-experienced one well and document the
remaining facet honestly. Premature solutions for corner cases
turn into design debt. The hint to look for: if the corner case
would require touching N collaborators (where N > 3) to fix
properly, defer it.

(Echo of ADR-022 lesson 2 — pure-rule first earned its keep.
Same principle: ship the common path well; the LLM-or-corner-case
path comes second.)

## Decision

Mark Phase 6.7 done. Update `CLAUDE.md` phase map. Sagittarius is
at v1.1.0 — the minor bump signals "Sagittarius now writes via MCP"
as a meaningful new capability.

The MCP bridge now exposes:
- 5 read tools (always, when the bridge is enabled)
- 9 write tools (when `mcpWriteEnabled`)
- 1 high-risk write tool — `delete_note` (when both
  `mcpWriteEnabled` AND `mcpHighRiskToolsEnabled`)

The diff-card-focus problem from ADR-023 lesson 2 is fully solved
for the user-not-at-Obsidian case. The in-app-chat-concurrent
case returns "retry shortly" — accepted limitation, follow-up
ADR if it bites.

Carry the two lessons above as guardrails for Phase 8+:

1. When phase decisions split into substrate/behavior/UX layers,
   slice across the layers. Each cut is a stable rest point.
2. Solve the realistic concurrency case well; defer the corner
   case until usage reveals it. A 5-line "retry" handler beats
   a 500-line refactor when the corner is rare.

## Follow-ups

- **Persistent queue across Obsidian restarts.** Currently the
  queue lives in memory; `clearAll` on unload rejects everything
  pending. If the operator closes Obsidian with proposals waiting,
  the MCP client sees clean rejection but the user has lost their
  review chance. A persistent queue would need its own ADR — open
  questions: proposals reference now-stale paths after reopen,
  what happens to the tool's promise (the apply closure is
  long-gone), how to surface "this proposal is from 3 days ago,
  do you still want it?". Defer until use reveals which of those
  matters.

- **Per-call WriteToolContext refactor.** The "in-app chat in
  progress, retry shortly" path is the accepted limitation from
  lesson 2. If real workflows hit it (operator chats while a long-
  running MCP write is also outstanding), file a new ADR proposing
  either AsyncLocalStorage-based scoping or constructor-injected
  per-call contexts. Until then, the singleton + retry path is
  fine.

- **Phase 8 — generative layer (ADR-026).** MVP at v1.2.0. Apply
  this ADR's lessons: when planning the slices, look for substrate
  / behavior / UX layers in the decisions. ADR-026 D1 (draft
  storage), D9 (diff-card reuse) feel like substrate; D2-D4
  (citation / drafting model / UI) feel like behavior; D5-D7 (UI
  surface, iteration, promotion) feel like UX. That's a clean
  three-slice cut.

- **OQ1 verification.** OQ1 was settled inline ("side-panel
  Approve skips the diff card"). If real usage suggests the
  diff card adds value despite the inline preview, the change
  is `gate.callback(proposal)` instead of
  `queue.respond(id, decision)` in `ExternalProposalsView.respond`.
  Note for future-me.
