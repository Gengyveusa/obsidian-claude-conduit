## Context

ADR-021 D1 shipped Phase 6.5 (the MCP bridge) as **read-only**:
`read_note`, `list_folder`, `search_vault`, `get_backlinks`,
`get_graph_neighborhood`. ADR-023 lesson 2 explicitly deferred the
write side because the **diff-card-focus problem** wasn't solved.

This ADR proposes the solution and stages Phase 6.7 тАФ exposing
Sagittarius's Phase 4 + v1.0.7 write tools (`create_note`,
`append_to_note`, `patch_note`, `rewrite_section`, `add_frontmatter`,
`move_note`, `rename_note`, `link_notes`, `file_asset`, `delete_note`)
to external Claude clients (Claude Desktop, Claude Code) over the same
MCP HTTP transport.

**The unlock:** ask Claude Desktop "add a follow-up to that meeting
note from last Tuesday" and it patches the note via Sagittarius, with
the diff card surfacing in Obsidian for your approval. Today that
round-trip requires switching apps + retyping context.

**The risk:** every Phase 4 write tool was designed assuming the user
is *looking* at Obsidian when the proposal pops. External calls
violate that assumption. If you're chatting in Claude Desktop and ask
for a patch, the diff card may render in a background Obsidian window
you don't notice тАФ leaving the MCP call hanging or, worse, getting
auto-rejected by a stale gate.

ADR-023 lesson 2 named this **the diff-card-focus problem**. The
decisions below propose three layers of mitigation:

  1. **Transport-level** тАФ block-with-timeout vs. queue-and-poll.
  2. **OS-level** тАФ system notification on proposal arrival, with
     "click to focus Obsidian."
  3. **UI-level** тАФ a status bar pill + side panel showing pending
     external proposals so they're never invisible.

The decisions also tighten the audit trail (ADR-021 D5 introduced
`source: 'mcp:<client>'`; this ADR makes that mandatory for every
write) and add a per-tool / per-client safety layer.

## Decisions

### D1 тАФ Tool scope: which write tools to expose

The Phase 4 write tools are not equally dangerous. Sketch the
blast-radius landscape:

| Tool | Blast radius | Reversibility |
|---|---|---|
| `create_note` | New file, no clobber | Trivial тАФ undo deletes the file |
| `append_to_note` | One file, additive only | Trivial тАФ undo restores body |
| `patch_note` | One file, arbitrary substitution | Trivial тАФ undo restores body |
| `rewrite_section` | One file, scoped rewrite | Trivial тАФ undo restores body |
| `add_frontmatter` | One file's YAML | Trivial |
| `move_note` | Cross-vault wikilink rewrites | Medium тАФ undo renames back |
| `rename_note` | Wikilink rewrites | Medium |
| `link_notes` | Two files | Trivial тАФ undo restores both |
| `file_asset` | New binary file | Trivial |
| `delete_note` | Destructive | Reversible via undo but loud |

**(a) All 10 tools.** Simple, consistent. Trusts the diff card.

**(b) All 10 minus `delete_note`.** Destructive operation needs explicit
in-app confirmation; we keep deletes manual-only via MCP.

**(c) Minimal тАФ `create_note` + `append_to_note` only.** Add tools as
trust accrues. Slowest expansion.

**(d) All 10, but `delete_note` requires a per-tool "high-risk" flag
in settings** (default off). User opts in when they want delete via
MCP.

`<DECISION D1: PROPOSED тАФ (d) all 10, with high-risk toggle for
delete_note. Other write tools land enabled by default; delete behind
a flag because the recovery path (undo) requires the user to be
present at Obsidian to even discover the deletion happened.>`

---

### D2 тАФ Diff-card-focus problem: response semantics

When an MCP write call arrives, the diff card needs the user. Four
shapes for how the MCP call relates to that approval:

**(a) Block until decision.** The MCP HTTP request hangs while the
diff card waits. Configurable timeout (default 60s); timeout returns
a structured `'timeout'` error. **Pro:** matches the MCP tool-call
contract (one-shot request тЖТ result). **Con:** if the user is away
from Obsidian, the call sits open and the external client thinks
Sagittarius is unresponsive.

**(b) Queue-and-poll.** The MCP call returns immediately with
`'queued'` + a `proposalId`. The external client polls
`get_proposal_status(proposalId)` until `'applied' | 'rejected' |
'expired'`. **Pro:** non-blocking, scales to long-deferred reviews.
**Con:** breaks the LLM tool-use loop тАФ Claude Desktop / Code expect
synchronous results to feed back into the conversation. Tool definition
would have to make the polling explicit.

**(c) Hybrid: block for short timeout, then auto-queue.** Block for 30s
(configurable); if no decision, return `'queued'` + `proposalId` and
the external client polls. **Pro:** synchronous when the user is
attentive, async when they aren't. **Con:** more state to manage.

**(d) Always queue, never block.** Every external write returns
`'queued'` immediately. The user reviews in their own time.
**Pro:** simplest mental model. **Con:** breaks the
read-write-and-confirm pattern most LLM tool flows expect.

`<DECISION D2: PROPOSED тАФ (c) hybrid, 30s block then auto-queue.
Tracks the "user is at the desk vs. user is away" reality without
requiring the user to learn polling-vs-blocking semantics. The
30s window is long enough for a "switching apps" round-trip but
short enough that an away user doesn't dangle the client.>`

---

### D3 тАФ OS-level notification on proposal arrival

When a proposal queues (D2 (c) timeout path), the user needs to know
without checking Obsidian.

**(a) None тАФ rely on Obsidian status bar.** Cheapest. **Con:**
invisible when Obsidian isn't focused.

**(b) Native notification via Electron's `Notification` API.** Pop a
system notification "Sagittarius: write proposal pending from
Claude Desktop. Click to review." Click тЖТ focus Obsidian +
auto-open the proposals panel.

**(c) macOS-only Dock badge.** Visual but click-to-respond is
unreliable.

**(d) Per-user toggle: settings.mcpWriteNotifyOnQueue (default on).**

`<DECISION D3: PROPOSED тАФ (b) + (d). Electron `Notification` on
proposal arrival with click-to-focus. Toggle on by default; user can
disable per ADR-019 D6 (everything intrusive is opt-out). On platforms
where Electron's notification API fails silently (rare; some Linux
WMs), we degrade to (a) without erroring.>`

---

### D4 тАФ UI surface for pending external proposals

The diff card was designed for one proposal at a time, in-flow. With
external MCP calls, multiple proposals can queue concurrently.

**(a) Modal diff card per proposal, FIFO.** Each pending proposal
opens a diff card; the user dismisses one and the next opens.
**Con:** modal-fatigue when batched.

**(b) Side panel "External proposals" view.** Companion to
Suggestions + Activity. Lists every pending external write with
Approve / Reject / Diff buttons inline. The diff card becomes a
preview-on-hover.

**(c) Status bar pill: "N external writes pending."** Clicks open
the side panel from (b). Doesn't surface the diff card automatically.

`<DECISION D4: PROPOSED тАФ (b) + (c). Side panel as the canonical
surface; status bar pill as the always-visible indicator. The diff
card per-proposal still opens (preserving ADR-016 D2: every write
gates on a diff card), but the side panel is the bird's-eye view
that prevents "where did that proposal go" disorientation.>`

---

### D5 тАФ Audit trail: `source: 'mcp:<client>'` on every write

ADR-021 D5 introduced the `source` field as optional. For Phase 6.7,
every MCP-driven write tags `source: 'mcp:<clientInfo.name>'` on:

  - the `ActivityEvent` (`write.committed`, `suggestion.applied`)
  - the `Transaction` (new `source` field on `Transaction`)
  - the transaction log

**(a) Optional + best-effort.** If `clientInfo.name` is absent in the
MCP `initialize` request, source = `'mcp:unknown'`.

**(b) Mandatory тАФ refuse `initialize` from clients without
`clientInfo.name`.**

**(c) Mandatory, but with a `mcpAllowedClients` fallback for testing
(`source = 'mcp:test'`).**

`<DECISION D5: PROPOSED тАФ (a). Best-effort matches MCP spec
permissiveness; refusing on missing client name would block valid
clients that omit it (rare but legal). `'mcp:unknown'` is a real
signal тАФ it tells the user "I don't know which app did this," which
is more useful than a refusal.>`

---

### D6 тАФ Per-client write permissions

Some users will trust Claude Desktop more than Claude Code (or vice
versa).

**(a) Single `mcpWriteEnabled` toggle.** All authenticated clients
either can write or can't. Per-client gating still happens at the read
layer (ADR-021 D9).

**(b) Per-client `mcpWriteAllowedClients: string[]`.** Subset of
the read-side allowlist. If empty, no client can write.

**(c) Per-client per-tool matrix.** Most flexible, most config burden.

`<DECISION D6: PROPOSED тАФ (a) + (b) combined. `mcpWriteEnabled`
master toggle, `mcpWriteAllowedClients` subset list. If
`mcpWriteAllowedClients` is empty AND `mcpWriteEnabled` is true, all
authenticated clients can write. Specifying a subset narrows.
Symmetric with ADR-021 D9.>`

---

### D7 тАФ Path scoping: where can external writes go?

A misconfigured client could ask for `patch_note` on
`THAD_MAN.md`. ADR-016 P1 already blocks vault-escape; this adds an
allowlist *within* the vault.

**(a) No path scoping.** Trust the diff card.

**(b) `mcpWritePathPrefixes: string[]` allowlist.** Default
`['10-Inbox/']`. Writes to paths outside this prefix list are
rejected at the MCP layer (before the diff card even shows).

**(c) Per-tool path scoping.** `create_note` in `_drafts/`, but
`patch_note` anywhere. Heavy.

`<DECISION D7: PROPOSED тАФ (b). Default scope = `['10-Inbox/']`,
matching the organization engine's watched-folders convention.
User widens. (a) is too permissive given how easy it is for an LLM
to misroute. (c) is over-engineering.>`

---

### D8 тАФ Auto-approve mode

For high-trust workflows (Claude Code editing its own scratch notes),
some users will want to skip the diff card per ADR-016 D2's reserved
`writeMode: 'auto'`.

**(a) Never auto-approve external writes.** Simple, safe.

**(b) Reuse `writeMode: 'auto'` from ADR-016 D2.** Same toggle covers
in-app and external.

**(c) Separate `mcpWriteAuto: boolean`.** External writes can
auto-approve even when in-app stays manual.

**(d) Per-tool auto-approve: `mcpAutoApproveTools: string[]`.**
e.g., `['append_to_note']` auto-applies but `'delete_note'` never.

`<DECISION D8: PROPOSED тАФ (a). Defer auto-approve to a later slice.
ADR-016 D2 reserved the path but never wired it; until we've shipped
auto-approve for in-app writes (where the user is staring at it),
shipping auto-approve for *external* writes is letting agents loose
on the vault. (d) is the eventual right answer but its threat model
needs its own ADR.>`

---

### D9 тАФ Rate limiting

A buggy or runaway external agent could spam writes. Phase 6.5's
auth is timing-safe, but a valid-token client with bad behavior
still has full access.

**(a) None.** Diff card is the rate limit.

**(b) Soft cap: N proposals/hour, beyond which return
`'rate-limited'`.** Default 30/hour. User-configurable.

**(c) Per-client soft cap.**

`<DECISION D9: PROPOSED тАФ (b). Default 30/hour is generous for human-
in-the-loop workflows but catches runaway loops. Per-client (c) is
over-engineering for v0.x; the global cap is fine.>`

### D10 тАФ Phase 6.7 milestone trigger

When does this ship?

**(a) v1.1.0** тАФ explicit minor bump signals "writing via MCP is now
supported," matching how v1.0.0 signaled "curator is now reachable."

**(b) v1.0.8** тАФ patch-level, treating MCP write-side as a v1.0
follow-up.

**(c) v1.x release once the diff-card-focus solution is battle-tested
on at least one external client.** Soft milestone.

`<DECISION D10: PROPOSED тАФ (a). v1.1.0. This is a meaningful
behavior change (external mutation of the vault) and a minor bump
makes the threshold visible to users who follow releases.>`

## Open question (no proposed decision)

**OQ1 тАФ Diff card vs. permissions: when D2 (c)'s 30-second window
expires and we auto-queue, does the proposal still need the diff
card on the eventual approval click, or can the queue-view "Approve"
button skip it?** Arguments for both:

  - Skip diff card from queue view: redundant (queue view shows the
    diff inline).
  - Require diff card: ADR-016 D2 invariant ("every write through the
    diff card") shouldn't be bypassed by UI shortcuts.

Surface for Thad's input.
