## Context

CLAUDE.md's phase map calls Phase 6 "Activity stream + MCP bridge — events, alerts,
digest." Spec §9 lists both as out-of-scope-until-Phase-6.

After shipping Phase 5, the actual pain points point one direction. The
ADR-018 lesson-3 retrospective said it directly: when something goes wrong
in the organization engine (or any future engine), the only way to see why
is DevTools console eval. That is not a sustainable diagnostic surface for
a tool you ship to other operators. **Activity Stream is the pain-driven
piece.** It also pulls in the diagnostics surface ADR-018 asked for as a
Phase 6 follow-up.

**MCP bridge is expansion-driven, not pain-driven.** Sagittarius works
inside Obsidian today. Exposing its tools to Claude Desktop / Claude Code
via Model Context Protocol is *nice-to-have* — opens a different surface
for the same vault — but doesn't fix anything broken. It also commits us
to a wire protocol with versioning, transport choice, and authn that we'd
rather not bolt on under the same phase header.

This ADR proposes splitting them. Phase 6 = Activity Stream + diagnostics
(this ADR). Phase 6b (or whatever number lands) = MCP bridge, its own ADR
when the user value justifies the protocol commitment.

---

## Decisions

### D1 — Scope: split Activity Stream from MCP bridge

Three options:

**(a) Split (recommended).** Phase 6 = Activity Stream + diagnostics only.
MCP bridge gets its own phase + ADR once the value-prop is sharper. This
ADR closes Phase 6 cleanly; MCP becomes "Phase 6.5" or rolls into Phase 7.

**(b) Bundle (per CLAUDE.md as-written).** Build both in one phase. Risks:
two substrates landing simultaneously, harder retrospective, MCP transport
decisions made under time pressure.

**(c) MCP-first.** Skip Activity Stream entirely; treat Obsidian's native
event log as sufficient. Risk: ADR-018 lesson 3 (diagnostics) stays
unaddressed. Future debugging stays in DevTools.

`<DECISION D1: ACCEPTED 2026-05-12 — (a) split Activity Stream from MCP bridge>`

---

### D2 — Event taxonomy

What gets recorded in the activity stream? Proposed event kinds:

| Kind | When | Payload |
|---|---|---|
| `index.built` | After `IndexCoordinator.ensureBuilt` | notesProcessed, chunksAdded, chunksSkipped, durationMs |
| `classifier.ran` | After every `classifyForRoute` / `classifyForMocAdd` | path, model, outcome (route/keep/moc-add/null), confidence, durationMs |
| `suggestion.enqueued` | When the queue accepts a new suggestion | suggestion summary (kind, path, target) |
| `suggestion.applied` | After diff card accepted + write committed | suggestion id, write tool used |
| `suggestion.rejected` | After diff card rejected | suggestion id |
| `suggestion.skipped` | User Skip-all or single Skip | suggestion id |
| `write.committed` | Any write tool path through transaction log | toolName, path |
| `write.undone` | Undo command | transaction id |
| `error` | Caught exception in watcher/classifier/write | source, message |

This is the v0.8.0 (first-slice) taxonomy. Future kinds (curator hits, MCP
calls, etc.) extend the union without breaking persistence.

`<DECISION D2: ACCEPTED 2026-05-12 — 9-kind taxonomy as proposed>`

---

### D3 — UX surface

Three options:

**(a) Dedicated side panel (recommended).** New `ActivityView` ItemView,
left-rail by default. Reverse-chronological list, filter chips by kind,
"Last 100 events" cap visible. Clicking an event with a path opens the
note.

**(b) Status-bar dropdown.** Click the Phase 5 status pill (or a new one)
→ flyout shows recent events. Compact but harder to scroll history.

**(c) Markdown digest note.** Auto-rewrite `_logs/activity-YYYY-MM-DD.md`
every N events. Native Obsidian discoverability; doesn't need a custom
view. Costs writes against the transaction log.

Sub-decision: any of these can also expose a `Sagittarius: Open
diagnostics` command that dumps the current watcher / queue / engine
state into the same view (the ADR-018 follow-up).

`<DECISION D3: ACCEPTED 2026-05-12 — (a) dedicated panel + diagnostics command>`

---

### D4 — Persistence + retention

Where does the event log live, and what's the size policy?

**(a) JSON file + rolling cap (recommended).** Persist to
`.obsidian/plugins/obsidian-claude-conduit/activity.json`. Cap at 1000
entries (rolling window — oldest dropped on overflow). Survives plugin
reload; tiny disk footprint; same pattern as `transactions.json` and
`suggestions.json`.

**(b) In-memory only.** Wipes on plugin reload. Simpler; useless for
debugging "what happened yesterday."

**(c) Append-only markdown note** (paired with D3 option c).

`<DECISION D4: ACCEPTED 2026-05-12 — (a) activity.json, rolling 1000-entry cap>`

---

### D5 — Alerts + digest

Spec mentions "alerts, digest" alongside the stream. Interpretation:

- **Alerts** — toast Notice on certain event kinds (error, suggestion.applied,
  index.built when notesProcessed > 0). Already partly exists (toasts from
  the watcher/index code). Phase 6 can consolidate the emission policy
  rather than building a new alerting system.
- **Digest** — a daily summary view: "Yesterday: 12 classifier calls, 3
  suggestions applied, 1 indexing pass, 0 errors."

Three options:

**(a) Stream only — punt alerts + digest.** Ship the activity stream view
in v0.8.0; alerts stay where they are; digest waits for a real user need.

**(b) Stream + digest, skip alerts** (recommended). Stream in v0.8.0;
digest as a "Last 24h" filter chip in the same view (cheap once the
event log exists). Skip alerts unification — premature.

**(c) Full bundle.** Stream + digest + alerts unification all in Phase 6.

`<DECISION D5: ACCEPTED 2026-05-12 — (b) stream + 24h digest filter; alerts untouched>`

---

### D6 — Rollout slicing

Three-version slice:

| Slice | Adds | Notes |
|---|---|---|
| **v0.8.0** — MVP | `ActivityLog` service + `activity.json` persistence + `ActivityView` panel + emission from watcher/classifier/write paths | One PR per substrate: emitter, persistence, view, integration |
| **v0.8.1** — Diagnostics | `Sagittarius: Open diagnostics` command + dump-state into the activity view | Closes ADR-018 lesson 3 |
| **v0.8.2** — Phase 6 close | "Last 24h" digest filter + Phase 6 retrospective ADR | Bulk skip on activity log (clear filtered) for the user who wants a clean slate |

`<DECISION D6: ACCEPTED 2026-05-12 — three-slice rollout v0.8.0 → v0.8.2>`

## Tool surface (binding)

No new write tools — Phase 6 is a read-and-display layer. The
`ActivityLog` is internal infrastructure consumed by the view. The
existing 9 write tools and the classifier emit events into it.

The agent does **not** get a `query_activity` tool in v0.8.x. If we
later want the LLM to introspect its own behavior ("what did I do
yesterday?"), that's a future ADR — read-only-event-exposure-to-LLM
is its own threat-model question.

`<DECISION D7: ACCEPTED 2026-05-12 — no new agent-facing tools in Phase 6>`

## Architecture sketch

```
src/
  activity/
    ActivityLog.ts           # JsonActivityLog : ActivityLog
    types.ts                 # ActivityEvent union (D2 kinds)
  views/
    ActivityView.ts          # ItemView, similar shape to SuggestionsView
  main.ts                    # construct + inject into classifier/watcher/write paths
```

Emission seam: every existing `console.warn('[sagittarius] ...')` in
watcher/classifier/write code becomes a dual call:
`this.activityLog.record({ kind: 'classifier.ran', ... })` *and*
the existing console line (kept for terminal debug).

Settings: one new toggle — "Record activity log" (default on). Off =
emitter no-ops + view shows a nudge to enable.

## Risks

| Risk | Mitigation |
|---|---|
| Event volume grows beyond 1000-cap on heavy-use days | Cap is per ADR D4; users with denser workflows can bump it via settings |
| `activity.json` corruption (concurrent writes, crashes) | Same pattern as `transactions.json` — atomic write via tmpfile + rename; reload tolerates empty/missing |
| View renders unresponsive at 1000 entries | Virtualize the list (render visible rows only); fall back to truncate at 100 if perf is bad |
| Privacy — event log contains paths | Stays local-only; never crosses the network; same threat model as `transactions.json` |
