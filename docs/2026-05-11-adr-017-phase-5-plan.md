## Context

Phase 4 (Write Layer) shipped as v0.5.0 end-to-end working. All 9 spec В§3.4 write tools are live with diff-first approval, per-turn transaction log, conflict detection, and an undo command. The next phase per the [CLAUDE.md phase map](../CLAUDE.md) is:

> 5 вҖ” Organization engine | auto-routing, MOC maintenance | future

**Different mental model than Phase 4.** Phase 4 was *reactive* вҖ” the user asks, the agent does, every action gates through a diff card. Phase 5 is *proactive* вҖ” the agent observes the vault, detects opportunities to improve structure, and *suggests* actions. The user batches accept/reject when they have time, not when the agent acts.

This ADR locks the design plan for Phase 5, the new infrastructure required (event watcher, suggestion queue, classifier service), and the rollout sequence from v0.6.0 (MVP) through v0.7.0 (Phase 5 close).

**Version-map note:** The CLAUDE.md phase map originally said "5 вҖ” Organization engine = v0.5." That target collided with Phase 4 close (which we shipped *as* v0.5.0). Phase 5 retargets to **v0.6.x вҶ’ v0.7.0**; CLAUDE.md is updated accordingly when this ADR is accepted.

Per [ADR-010 В§4](2026-05-04-sagittarius-build-process.md), no Phase 5 code lands before this ADR is signed off.

## Decisions

### D1 вҖ” Trigger model

**Hybrid: vault events for create + manual sweep + a "check inbox now" command.**

```
Obsidian `vault.on('create', file)` fires вҶ’ if file is in a watched folder,
  queue an organize-task for that file (debounced 5s after the user stops typing).

User runs `Sagittarius: organize inbox now` вҶ’
  scan every file in watched folders that isn't already in the suggestion queue.

[Optional] Background sweep every N minutes (default off; user opt-in via setting).
```

**Rationale:**
- Events are responsive вҖ” new note created вҶ’ suggestion appears in seconds.
- Manual sweep handles the catch-up case (e.g., user pasted 20 notes from another app at once).
- Background sweep is opt-in because most users don't want a periodic process running.
- 5s debounce prevents thrashing on rapid-fire edits to the same file.

**Rejected:**
- *Pure event-driven*: misses the catch-up case.
- *Pure periodic*: feels laggy for fresh notes.
- *Manual only*: defeats the "proactive" framing of the phase.

`<DECISION D1: ACCEPTED 2026-05-11>`

### D2 вҖ” Suggestion UX surface

**Dedicated sidebar panel: "Sagittarius вҖ” Suggestions" (`SuggestionsView`). Separate from the chat panel.**

UI structure:
- One row per pending suggestion: `[icon] description  [Apply] [Skip] [Defer]`
- Description shows the proposed action in plain English: "Move `10-Inbox/Hangar-meeting-2026-05-11.md` to `70-Memory/meetings/`. Reason: 4 of 5 similar notes live there."
- `Apply` runs the proposed write through the standard diff card from Phase 4 (so the user still confirms).
- `Skip` removes the suggestion from the queue without applying.
- `Defer` keeps it in the queue but moves it to the bottom.
- Header shows count of pending + a `Clear all` action.

**Why a separate panel, not chat-card or notice:**
- Chat-card was designed for 1:1 reactive flow (user asks вҶ’ agent proposes). Mixing proactive suggestions in there drowns conversation.
- Notice popups are too intrusive вҖ” N suggestions could mean N popups firing in rapid succession.
- A panel matches the user's mental model: "the queue of things Sagittarius noticed."
- Ribbon icon (lightbulb or list) with a badge for unread suggestions provides ambient awareness without modal interruption.

**Rejected:**
- *Notices*: rejected per ADR-016 D2 (same reasoning).
- *Chat-card embed*: mixes mental models.
- *Settings-tab list*: too buried; users won't check.

`<DECISION D2: ACCEPTED 2026-05-11>`

### D3 вҖ” Suggestion types for v0.6.x

**v0.6.x ships two suggestion kinds; later versions extend.**

1. **`route` вҖ” folder routing for inbox notes.**
   - Inputs: a note's path + body + frontmatter.
   - Decision: should this note stay in `10-Inbox/` or move to a more-permanent folder?
   - Apply: routes through `move_note` (which uses Obsidian's `fileManager.renameFile` so wikilinks auto-update).

2. **`moc-add` вҖ” add a note to an existing Map-of-Content note.**
   - Inputs: a note path + the path of a MOC that might want it.
   - Decision: is the new note related-enough to belong on this MOC?
   - Apply: routes through `link_notes` (insert a `[[note]]` under the MOC's anchor section).

**v0.7.0 may extend with:**
- `frontmatter-suggest` (add common frontmatter fields based on note shape)
- `moc-create` (detect cluster of related notes with no MOC, suggest creating one)
- `tag-suggest` (propose tags based on similar notes)

`<DECISION D3: ACCEPTED 2026-05-11>`

### D4 вҖ” Routing intelligence

**LLM-as-classifier with retrieval-grounded context. Sonnet default (Thad override, 2026-05-11).**

For each `route` suggestion:

```ts
// In OrganizationClassifier service (new):
1. Read the note's content + frontmatter
2. Use the existing RetrievalLayer to find K=5 most-similar existing notes
3. Compose a short prompt:
   "Note title: <basename>
    Note body (first 500 chars): <body>
    Similar existing notes and their folders:
    - 70-Memory/meetings/foo.md (similarity 0.84)
    - 70-Memory/meetings/bar.md (similarity 0.79)
    - 21-Agents/concierge.md (similarity 0.61)
    ...
    Where should this note live? Reply with ONE folder path or KEEP."
4. Call Claude (Sonnet default per override below) with the prompt
5. Parse response вҶ’ produce a Suggestion
```

**Why LLM over rules:**
- The vault structure is bespoke per-user; no fixed rule schema will fit.
- Claude is good at noticing thematic clusters from a small retrieval sample.
- The classifier includes the user's THAD_MAN.md constitution in its system prompt so routing respects vault conventions.

**Model choice вҖ” Sonnet over Haiku:**
- *Original recommendation was Haiku for cost.* Thad overrode (2026-05-11): the classifier needs to internalize THAD_MAN.md conventions (e.g., "FortressFlow lives in 50-FortressFlow/", "ADRs go in 22-Decisions/") and reason about which numbered folder a note belongs in. Sonnet's stronger instruction-following pays off here вҖ” a wrong route is a noisy suggestion the user has to dismiss, eroding trust in the queue. The ~3-5Г— cost bump vs. Haiku is acceptable because watched-folder scoping keeps total volume low (~10-50 classifications per day for a typical inbox).
- Setting field stays a dropdown вҖ” user can downgrade to Haiku if they prefer cost over quality. Opus is also available for full nuance; left as an opt-in.

**Cost mitigation:**
- Only run for notes in watched folders.
- Don't re-suggest the same note within N hours.
- Confidence-thresholded queue вҖ” low-confidence outputs hidden behind a toggle.

**Confidence threshold:**
- Classifier emits a confidence score (0..1) вҖ” Claude is asked to also return one in its JSON response.
- Only surface suggestions with confidence вүҘ `organizationMinConfidence` (default 0.6).
- Below threshold вҶ’ suggestion stored but hidden behind a "show low-confidence" toggle.

**Rejected:**
- *Pure rules / regex*: too brittle.
- *Local ML classifier*: too heavy for the plugin bundle.
- *Frontmatter-only*: works for notes that already have tags; useless for raw inbox dumps.

`<DECISION D4: ACCEPTED WITH OVERRIDE 2026-05-11 вҖ” LLM-classifier + retrieval + Sonnet default + 0.6 threshold>`

### D5 вҖ” User control / settings

**New settings (extending `SagittariusSettings`):**

```ts
// Phase 5 organization engine
organizationEnabled: boolean;            // default false вҖ” opt-in. Won't watch until user flips this.
organizationWatchedFolders: string[];    // default ['10-Inbox/']
organizationClassifierModel:             // default 'claude-sonnet-4-6' (per D4 override)
  'claude-haiku-4-5-20251001' | 'claude-sonnet-4-6' | 'claude-opus-4-7';
organizationMinConfidence: number;       // 0..1, default 0.6 вҖ” threshold for surfacing in queue
organizationSweepIntervalSec: number;    // default 0 (manual only); 3600 = hourly background sweep
organizationMocFolders: string[];        // folders where MOCs live (e.g., '22-Decisions/'); empty disables moc-add
```

**Defaults rationale:**
- `organizationEnabled: false` вҖ” Phase 5 is a meaningful behavior change. Don't opt users in silently.
- Watch only `10-Inbox/` by default вҖ” that's where unstructured drafts land.
- Sonnet default (per D4 override) вҖ” better routing quality vs. Haiku at modest cost. Users can downgrade to Haiku via the dropdown if they prefer cheaper-but-noisier suggestions.
- 0.6 threshold вҖ” biased toward surfacing more suggestions than fewer; users can tighten.
- Sweep off by default вҖ” explicit "organize inbox now" command is enough for most.

**Settings UI:**
- New "Organization (Phase 5)" section in the settings tab.
- Toggle for `organizationEnabled` вҖ” when off, no watcher fires, no classifier calls happen, no queue items get added.
- All other fields are inputs (text / dropdown / number).

`<DECISION D5: ACCEPTED 2026-05-11 (with D4 override applied to organizationClassifierModel default)>`

## Tool surface (binding)

Phase 5 doesn't add new agent-callable tools. The classifier runs **outside** the ConduitAgent loop вҖ” it uses Anthropic's SDK directly with a fixed prompt. The agent's tool registry is unchanged from v0.5.0.

The user-facing surface is:
- Command palette: `Sagittarius: organize inbox now` вҖ” runs sweep + classifier
- Command palette: `Sagittarius: open suggestions panel` вҖ” opens the SuggestionsView
- Ribbon icon (lightbulb) with badge for pending count
- The suggestions panel itself, with Apply/Skip/Defer per row

**Implementation note on Apply.** Two options for wiring "user clicked Apply":
- (a) Call the write tool directly from the panel handler (bypassing the agent). Faster, fewer moving parts, but no LLM in the loop.
- (b) Inject a synthetic user message into the chat agent ("Move foo.md to bar/") and let the normal agent loop handle it. Slower, more cost, but reuses the entire Phase 4 approval pipeline.

`<DECISION D8: ACCEPTED 2026-05-11 вҖ” (a) direct write>`

Recommendation: (a) for performance + simplicity. The diff card from Phase 4 still fires (because all write tools route through `gate.request()`), so the user still confirms before any file changes. The only thing missing in (a) is "Claude narrates what it did" вҖ” which the suggestion queue already shows in plain English anyway.

## Risks

| Risk | Mitigation |
|---|---|
| Classifier cost spirals on a noisy inbox | Default to Haiku + dedupe by path within N hours + watched folders only |
| User finds suggestions annoying | Opt-in by default; clear "disable organization" toggle |
| Suggestion queue grows unbounded | Cap at 200 entries with FIFO truncation (same pattern as TransactionLog's 1000-cap) |
| Classifier picks wrong folder repeatedly | User can `Skip` to dismiss; pattern of skips on similar notes can feed a future v0.7 "learned skip" feature |
| Vault event noise (rapid edits) | 5s debounce per file |
| Notes outside watched folders accidentally moved | Watcher filters by `organizationWatchedFolders` вҖ” only files inside trigger classification |
| Apply triggers a write conflict | Goes through the existing Phase 4 write tools, which already have ConflictDetector + the diff card surfaces conflicts cleanly |
| User accepts a bad move, can't undo | All Apply writes go through the transaction log вҶ’ `Sagittarius: Undo last write transaction` works as in Phase 4 |

## Out of scope (Phase 6+)

- Cross-vault organization (multi-vault routing rules)
- Learning from user skips ("you've skipped 5 similar suggestions; should I stop showing these?")
- Tag autocomplete / tag taxonomy management
- Conversation memory (linking related chat turns to MOCs)
- Cited generative drafts (Phase 8)
- CLAUDE.md reader/writer / dossiers (Phase 9)
