## Context

Phase 3 (Read Layer) shipped as v0.2.6, end-to-end working. Per the phase map in CLAUDE.md and the binding v0.1 spec (§4 "Out of scope" → Writes of any kind = Phase 4), the next phase introduces **diff-first writes** with a **transaction log** and **undo**. Spec §3.4 names nine tools:

| Tool | Purpose |
|---|---|
| `create_note` | New markdown file at a vault-relative path. |
| `patch_note` | Apply a structured patch to an existing note's body. |
| `append_to_note` | Append content to a note (or create-if-missing variant TBD). |
| `move_note` | Move a note to a new folder (preserves links per Obsidian's metadata cache). |
| `rename_note` | Change a note's filename (preserves links). |
| `link_notes` | Insert a `[[wikilink]]` reference between two notes. |
| `add_frontmatter` | Add or modify a YAML frontmatter field. |
| `rewrite_section` | Replace the body of a specific `## Section` header. |
| `file_asset` | Save a binary attachment (image, PDF) into the vault's attachments folder. |

This ADR locks the design plan for those nine tools, the surrounding infrastructure (transaction log, approval UX, conflict detection, path-traversal validation), and the rollout sequence from v0.3.0 (MVP) through v0.5.0 (Phase 4 close).

Per [ADR-010 §4](2026-05-04-sagittarius-build-process.md), no Phase 4 code lands before this ADR is signed off.

---

## Decisions

Each block below carries my recommendation. Open items are flagged `<DECISION: ...>` — Thad's call.

### D1 — Diff representation

**Internal:** structured patch ops. Each tool emits one or more of:
```ts
type PatchOp =
  | { kind: 'replace', startLine: number, endLine: number, content: string }
  | { kind: 'insert',  afterLine: number, content: string }
  | { kind: 'delete',  startLine: number, endLine: number }
  | { kind: 'create',  content: string }              // create_note only
  | { kind: 'rename',  from: string, to: string }     // move/rename only
  | { kind: 'frontmatter', op: 'set'|'delete', key: string, value?: unknown };
```

**User-facing:** render each op as a unified diff in the chat panel, +/- styled like `git diff`. Same content the agent's tool_use produced, just visualized.

**Rationale:**
- Structured ops are atomic and inverse-able (essential for undo).
- Cleaner transaction log than whole-file before/after blobs.
- Unified diff is universal for the user; renders cleanly in markdown.

**Rejected:**
- Whole-file before/after: large UI surface, verbose transaction log, hard to invert cleanly.
- Naive text-search-and-replace: brittle to whitespace, ambiguous on duplicate strings.

`<DECISION D1: accept structured ops + unified-diff visualization, OR override>`

---

### D2 — Approval UX

**Inline in chat, per-tool-call, always required (no auto-apply in Phase 4).**

After Claude calls a write tool, the tool **does not execute immediately**. It records a *proposal* and returns control to the agent loop with a result that says "proposed; awaiting user approval." The chat panel renders a diff card directly under the message, with [✓ Accept] and [✗ Reject] buttons. The user clicks one; the tool result is resolved (or the next tool in the message blocks until decided).

If accepted, the patch applies, the transaction log records it, and the tool returns `{ status: 'applied', ... }` to the agent. If rejected, the tool returns `{ status: 'rejected', reason: 'user rejected' }` and the agent can revise.

**A user setting (`writeMode: 'review' | 'auto'`) is reserved but defaults to `'review'`.** `'auto'` ships in v0.5+ after Phase 4 is battle-tested.

**Rationale:**
- Inline keeps the conversational mental model — accept right where you see the proposal.
- Per-tool-call lets the user accept some changes and reject others within one agent turn.
- Hard-required approval mirrors the "diff-first writes" framing in every prior doc (CLAUDE.md, README, ADR-007).

**Rejected:**
- Quarantine queue / batch review: better for Phase 7 (curator's proactive suggestions). Wrong fit for conversational writes.
- Auto-apply with undo: too easy to lose work if the agent misunderstands. Spec §7 threat model treats vault corruption as the central Phase 4 risk.

`<DECISION D2: accept inline + per-tool + always-required, OR override>`

---

### D3 — Granularity

**Atomicity = per-tool-call (one file at a time). Undo = per-agent-turn (groups all approved tools from one message into one undo unit).**

Each write tool is atomic at the file level. If a tool's apply step fails midway (rare — we'd have to crash between mkdir and write), the file is in whatever state Obsidian's `DataAdapter.write()` left it. We don't promise multi-file transactions.

The **transaction log** groups all *approved* tools from a single agent turn into one logical transaction. `Undo last transaction` reverts every approved write from that turn, in reverse order.

**Rationale:**
- Per-tool atomicity = "the unit that succeeded or failed is exactly the unit I approved." Matches user intuition.
- Per-turn undo = "Claude proposed 3 changes; I accepted all 3; one undo reverts the whole thing." Matches the conversational frame.

**Rejected:**
- Per-tool undo: noisy. Three clicks to undo one logical change.
- Per-session undo: too coarse. User wants surgical control.

`<DECISION D3: accept per-tool atomicity + per-turn undo, OR override>`

---

### D4 — Conflict handling

**Optimistic concurrency with mtime + SHA-256 hash.**

When a write tool reads a file (`patch_note`, `rewrite_section`, `add_frontmatter`, `append_to_note`'s create-if-missing branch), it captures `{ mtime, sha256 }` at read time. When the user approves the proposed write, the tool re-stats the file and re-hashes its current contents. If either differs from the read-time snapshot, the apply step **aborts and surfaces a conflict error** to the agent: `"File modified by another writer (likely user edit). Re-read and re-propose."`

The agent then re-runs the read tool and re-proposes the patch against the new state. The user sees this as a re-rendered diff card.

**For tools that don't read** (`create_note` only — when target path is empty), no conflict check is needed. `exists()` is enough.

**Rationale:**
- Pessimistic locking is wrong in Obsidian — the user is *always* editing.
- Last-writer-wins is the documented v0.1 antimemory bug we're trying to avoid (spec §7).
- mtime alone is fragile (subsecond writes, mtime resolution); hash makes it robust.

**Rejected:**
- Lock files: no clean way to lock in Obsidian's renderer model.
- YOLO: would silently overwrite user edits. The threat model explicitly calls this out.

`<DECISION D4: accept optimistic + mtime+SHA-256, OR override>`

---

### D5 — Rollout order

**Four-version slice through v0.5.0 (Phase 4 close):**

| Version | Scope | Why first/last |
|---|---|---|
| **v0.3.0 — Foundation MVP** | Transaction log + diff render + approval UX scaffolding + `create_note` + `append_to_note` | These two are pure-creation, no read-then-modify. Validates the whole pipeline end-to-end with the simplest tools. |
| **v0.3.x — Read-modify-write trio** | `patch_note` + `rewrite_section` + `add_frontmatter` + **conflict detection** | First time we need mtime+hash. Lands together so the infra is exercised. |
| **v0.4.0 — Undo command** ([amended 2026-05-11](#amendment-2026-05-11-undo-moved-to-v040)) | `adapter.delete()` + `undo_last_transaction` command + transaction log replayer | Validates the inverse-op shape against real on-disk data. Ships safety net **before** the high-blast-radius graph tools. |
| **v0.4.1+ — Graph manipulations** | `move_note` + `rename_note` + `link_notes` | Need to coordinate with Obsidian's metadata cache to update backlinks. Trickier than body patches. |
| **v0.5.0 — Close** | `file_asset` + polish (settings toggle for `writeMode`, status indicator in chat for pending proposals) | `file_asset` is binary-only territory. `writeMode` setting ships as the public Phase 4 release. |

**Prereqs (must land before any write tool):**
- Path-traversal validation helper (`assertInVault`) — flagged in ADR-015 follow-ups, spec §7 binding. Ships in v0.3.0 PR 1.

**Rationale:**
- MVP-first reduces risk — we ship the infra (transaction log, approval UX, diff rendering) with the simplest possible tools to validate before complicating.
- The read-modify-write trio in v0.3.x exercises conflict detection at the right granularity.
- Graph manipulations are last in the "edits" pile because Obsidian's metadata cache integration is the tricky bit; we want to be confident in the rest first.
- **(Amended 2026-05-11)** Undo lands as v0.4.0, *before* the graph trio — see amendment below.

**Rejected:**
- Ship all 9 at once: too big a PR; impossible to test incrementally.
- Strict spec order: the spec names the tools in §3.4 but doesn't impose an order. Ours is risk-stratified.

`<DECISION D5: accept four-version slice, OR override with a different order>`

## Phase 4 prereqs (from [ADR-015](2026-05-10-adr-015-vault-adapter-audit.md))

### P1 — Path-traversal validation

Every write tool must validate that the target path stays inside the vault:

```ts
export function assertInVault(adapter: VaultAdapter, vaultRelativePath: string): void {
  if (vaultRelativePath.includes('..') || vaultRelativePath.startsWith('/') || vaultRelativePath.startsWith('~')) {
    throw new VaultPathTraversalError(vaultRelativePath);
  }
  // Defense in depth: also check that the resolved path's prefix matches the vault root
  // (real-path resolution would catch symlinks too, but Obsidian's adapter doesn't expose them).
}
```

Lives in `src/agent/validation.ts` (new file). Every write tool imports it and calls before any `adapter.write()` / `adapter.writeBinary()` / `adapter.mkdir()`. Test surface: at least 12 cases (legitimate paths, parent-dir traversal, absolute paths, tilde, NULL bytes, unicode quirks).

Spec §7 is **binding**; this prereq is non-negotiable. Lands in **v0.3.0 PR 1** (before any write tool).

### P2 — Write-conflict detection

Implementation is part of D4 above. The helper is:

```ts
export class ConflictDetector {
  async snapshot(adapter: VaultAdapter, path: string): Promise<FileSnapshot>;
  async verifyUnchanged(adapter: VaultAdapter, path: string, snapshot: FileSnapshot): Promise<void>;
  // throws WriteConflictError if mtime or hash drifted
}
```

Lives in `src/writes/ConflictDetector.ts`. Lands in **v0.3.x** (with the first read-modify-write tool, `patch_note`).

## Tool surface (binding — agents read this)

Tool input schemas, all using Zod for validation per existing ToolRegistry pattern:

### create_note
```ts
{ path: string, content: string, createParentDir?: boolean = true }
```
Refuses if path exists. Refuses if path fails `assertInVault`. Returns the created path.

### patch_note
```ts
{ path: string, ops: PatchOp[], expectedMtime: number, expectedHash: string }
```
Applies ops in order. Bails if conflict (mtime+hash). Returns new mtime.

### append_to_note
```ts
{ path: string, content: string, createIfMissing?: boolean = false, expectedMtime?: number, expectedHash?: string }
```
Adds at end with a leading blank line. If `createIfMissing` and not exists, behaves like `create_note`.

### move_note
```ts
{ fromPath: string, toPath: string }
```
Uses `app.fileManager.renameFile()` so Obsidian's metadata cache updates wikilinks automatically.

### rename_note
```ts
{ path: string, newName: string }   // newName has no extension; .md added
```
Same metadata cache integration as move_note.

### link_notes
```ts
{ fromPath: string, toPath: string, anchorInFrom?: string, expectedMtime: number, expectedHash: string }
```
Inserts a `[[toPath]]` reference into fromPath. If `anchorInFrom` is given, inserts after that text; otherwise at end of body.

### add_frontmatter
```ts
{ path: string, key: string, value: string | number | boolean | string[], expectedMtime: number, expectedHash: string }
```
Adds or updates a frontmatter field. Creates the frontmatter block if missing.

### rewrite_section
```ts
{ path: string, sectionHeader: string, newBody: string, expectedMtime: number, expectedHash: string }
```
Replaces the body content under `## sectionHeader` (matched by exact heading text + level). Preserves heading itself.

### file_asset
```ts
{ filename: string, base64Content: string, folder?: string }
```
Writes a binary asset to `folder || (default attachments folder from Obsidian settings)`. Refuses if filename fails `assertInVault`.

`<DECISION D6: accept the tool surface above, OR flag any field changes>`

## Risks

| Risk | Mitigation |
|---|---|
| User loses unsaved work to a Claude write | Mandatory approval per D2; conflict detection per D4; transaction log + undo per D3. |
| Prompt injection causes a malicious path | `assertInVault` per P1; spec §7 binding. |
| Metadata cache gets out of sync after move/rename | Use `app.fileManager.renameFile()` not `adapter.write()` so Obsidian handles cache update. |
| User edits during approval | Optimistic check at apply time, not at propose time. Conflict surfaces cleanly. |
| Transaction log grows unbounded | Rotate / cap at 1000 entries with rolling truncation. JSON file under `.obsidian/plugins/obsidian-claude-conduit/transactions.json`. |
| Binary writes (file_asset) bloat the index | `excludePathPrefixes` already covers attachments folder; verify in v0.5. |

---

## Out of scope (Phase 5+)

- Auto-mode (`writeMode: 'auto'`) — settings field reserved but UI ships disabled.
- Multi-file atomic transactions — per D3, per-tool atomic is the contract.
- Vault-wide undo beyond the transaction log — once a transaction is undone, redo is not supported.
- Proactive curator suggestions (Phase 7).
- Cited generative drafts (Phase 8).
- AI-suggested merges when a conflict is detected — Phase 4 just surfaces conflicts; resolution is the user's job.

## Amendment 2026-05-11 — Undo moved to v0.4.0

After shipping v0.3.0 (foundation MVP) and v0.3.1 (read-modify-write trio + `ConflictDetector`), we pulled undo forward from v0.5.0 to v0.4.0, before the graph trio. Thad signed off on the reorder.

**Reasons:**

1. **The transaction log has been writing inverse ops for two releases but nothing has replayed one yet.** Until a consumer actually reads + applies an `InverseOp`, we don't know the inverses are correct. Shipping undo is the end-to-end validation of the entire write layer — including the v0.3.x trio's `write-file` inverse with the prior content.
2. **Risk stratification.** Graph tools (`move_note`, `rename_note`) coordinate with Obsidian's metadata cache to update every wikilink in the vault — high blast radius. Shipping the safety net first means we deploy undo against the (well-tested, small-radius) v0.3.x mutations before exposing tools that could touch hundreds of files.
3. **Scope is small.** One new command (`Sagittarius: Undo last write transaction`), a replay loop, and a `delete-file` adapter method. Probably one PR plus a small `VaultAdapter` extension.
4. **Both v0.4 and undo need a `VaultAdapter` extension anyway** — `delete()` for undo, `renameFile()`-via-`app.fileManager` for the graph trio. Same shape of architectural work.

**Reorder:**
- ~~v0.5.0 ships `file_asset` + undo command + `writeMode` setting~~
- **v0.4.0** ships undo command (this amendment).
- v0.4.1+ ships the graph trio (unchanged in spirit, version bumped).
- v0.5.0 ships `file_asset` + `writeMode` setting + Phase 4 close (was originally bundled with undo; now slimmer).

**What this doesn't change:**
- D1-D4 are untouched. The diff-rendering, approval UX, atomicity unit, and conflict-handling decisions all still apply to the graph trio and `file_asset` when they ship.
- D6 (tool surface) is unchanged — all 9 tools' contracts stay as drafted.
- Phase 5+ out-of-scope items (auto-mode, multi-file atomicity, redo, AI-suggested merges) are still out.
