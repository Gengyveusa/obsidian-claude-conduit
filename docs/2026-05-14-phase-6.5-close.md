## Context

Phase 6.5 (MCP bridge) opened with ADR-021 on 2026-05-13 and closes
today with v0.9.2. The phase took five PRs over one session (#64 →
#72) and shipped against the v0.8.2 baseline. What landed:

- **v0.9.0** — four-PR slice:
  - PR 1 (#64): `@modelcontextprotocol/sdk@1.29.0` dependency,
    bearer-token auth helpers (generate / hash / timing-safe compare
    / parse / verify), `McpServer` scaffold class, 4 new settings
    (`mcpEnabled` / `mcpPort` / `mcpToken` / `mcpAllowedClients`).
  - PR 2 (#66): `HttpListener` — Node `http.createServer` wrapper
    bound to `127.0.0.1` (D4), Bearer-auth gating (D3), 1 MiB body
    cap, OPTIONS preflight, JSON / 405 / 400 / 413 / 500 / 503 error
    paths. `McpServer` composes the listener with a stub echo handler.
  - PR 3 (#68): hand-rolled JSON-RPC dispatcher (`McpHandler` +
    `JsonRpc.ts` + `McpToolAdapter.ts`). Three methods: `initialize`,
    `tools/list`, `tools/call`. Five read-only tools exposed per
    D1 (c) + D9 (`read_note`, `list_folder`, `search_vault`,
    `get_backlinks`, `get_graph_neighborhood`); write tools stay
    registered with `ToolRegistry` but the allowlist filters them
    at the boundary.
  - PR 4 (#71): `main.ts` lifecycle (`refreshMcpServer()`,
    `generateMcpToken()`, `onunload` teardown), MCP settings UI
    section, activity-stream `source:` field (D5). McpHandler
    captures `clientInfo.name` during `initialize` and tags
    subsequent tool-call events as `source: mcp:<client>`.
- **v0.9.1** (#72) — polish: `MCP` filter chip in `ActivityView`,
  `Sagittarius: Test MCP connection` command (pings without a token,
  expects 401 = healthy + auth gating).
- **v0.9.2** (this PR) — close: this retrospective ADR + version
  bump + release.

Test count grew from 606 (Phase 6 close) to 766, all green.
Substrate adds: `src/mcp/` (5 files — auth, HttpListener, McpServer,
McpHandler, McpToolAdapter, JsonRpc). Settings gained 4 fields. No
new agent-facing tools (D10 invariant; only existing tools were
re-exposed).

## The two lessons of Phase 6.5

### 1. "Use the official SDK" doesn't always mean using its transport

ADR-021 D8 chose `@modelcontextprotocol/sdk` per the ADR-014 lesson
("bundle the official SDK so URL evolution is the SDK's problem").
We did bundle it (~200KB hit, accepted). But when wiring tool
registration in PR 3, the SDK's `Server` + `Transport` machinery
wanted to own the request/response loop in a way that didn't fit
our existing `HttpListener` (which already owned auth + body parse).
The integration would have required either:
  - rewriting `HttpListener` to use the SDK's `streamableHttp`
    transport (drops our auth seam), or
  - building a custom `Transport` that holds per-request state
    (~150 lines of glue for behavior we wrote in 30).

We hand-rolled JSON-RPC dispatch for v0.9.0 instead. The protocol
is tiny — three methods, two response shapes, one error format —
and the SDK's types still provide schema validation for shapes.
Total dispatch code: ~150 lines including comments. Easy to test
end-to-end against real HTTP.

**Lesson:** "use the official SDK" is a contract about protocol
stability, not a mandate to use every layer the SDK ships. When the
SDK's higher-level abstractions don't fit the integration point,
use the SDK for its types + canonical schemas and own the
request-response cycle yourself. ADR-014 said "bundle the official
SDK so URL evolution is the SDK's problem" — that's the right
contract, and using the SDK's types satisfies it. Future ADRs that
say "use the SDK" should clarify: are we binding to the protocol or
to the SDK's runtime?

### 2. Tokens that are "shown once" need a place to belong

PR 4's `generateMcpToken()` mints a 32-byte random token, hashes it,
persists the hash, returns the raw value. The settings UI shows
the raw value in a 30-second `Notice`. The user copies it to their
external client's config; the raw value is then unrecoverable.

This *works*, but lives at the edge of "is this acceptable UX?" If
the user dismisses the Notice before copying, they have to
regenerate (invalidating the previously-distributed token). The
30-second timer is arbitrary. There's no "this token is currently
configured at: ~/.claude/mcp.json" hint.

We shipped it anyway because:
  - the alternative (storing the raw token in settings) is worse —
    settings.json gets backed up, synced, copied; tokens leak
  - the regeneration path is one click; not catastrophic
  - it's correct for the threat model (one operator, one machine,
    paranoia-by-default)

**Lesson:** "show once, never again" is acceptable for one-operator
local tools when the regenerate cost is low. But the next time we
ship a similar pattern (API keys, OAuth refresh tokens, encryption
keys), we should also: surface a *current state* indicator
("token configured" / "no token") in settings; auto-clear the Notice
on a click rather than a timer; document where the raw value
should end up. Phase 7's curator already does the first of these
("token configured" check in settings); the others remain
follow-ups.

## Decision

Mark Phase 6.5 done. Update `CLAUDE.md` phase map. The MCP bridge
is functional end-to-end: `Sagittarius: Test MCP connection` returns
401 (healthy), Claude Desktop / Claude Code can configure
`http://127.0.0.1:8765/` with the bearer token and call any of the
5 read-only tools.

D1 (c) deferred write-side tool exposure. **Defer further** —
write-side over MCP raises three issues we haven't solved:

  1. **Diff card focus problem.** External tool call triggers diff
     card in Obsidian; if Obsidian isn't focused, the external
     client's call hangs until the user notices. Need timeout +
     "background diff" UX (per ADR-021 risks table).
  2. **Activity-stream double-recording.** Phase 6's TransactionLog
     already emits `write.committed`; MCP-driven writes would emit
     twice (once via TransactionLog, once via McpHandler). Need to
     dedupe or attribute correctly.
  3. **Token leak blast radius.** Read-side leaks expose vault
     contents (bad). Write-side leaks let an attacker corrupt or
     destroy the vault (catastrophic). Higher bar for the auth
     model.

Each is solvable; none is solved tonight. Leave write-side as a
future phase (Phase 6.7 or Phase 9+ alongside MCP-driven memory).

Carry the two lessons above as guardrails for Phase 7+:

1. SDK adoption is about protocol stability, not transport
   ownership. State which layer the ADR is binding to.
2. "Show-once" credential UX needs a current-state indicator + a
   click-to-dismiss; not just a timer.

## Follow-ups (carry into Phase 7 / future MCP work)

- **Write-side MCP exposure** — deferred. Revisit with a dedicated
  ADR once Phase 7 closes and the diff-card-focus problem has a
  Phase-6.5-style answer (probably a "pending external write" panel
  with explicit approval).
- **MCP `clientInfo` allowlist enforcement** — `mcpAllowedClients`
  field exists in settings but isn't enforced in the handler yet
  (PR 4 captured it for activity attribution but not for
  access control). Add to v0.9.x or Phase 7 polish window.
- **Port-conflict UX** — current behavior surfaces the error message
  via Notice. Could be friendlier (suggest the next available port,
  auto-retry on `EADDRINUSE` with port + 1).
- **`Sagittarius: Test MCP connection` should also verify token
  validity** — current test only verifies reachability + auth
  gating. A "test with current token" variant would need either
  (a) re-prompting for the raw value or (b) keeping the raw value
  in memory after generation for the session.
- **Streaming responses** — v0.9.0 is request/response only. MCP
  spec supports SSE notifications; future ADR if tool calls need
  progress streaming (e.g., large search results).
